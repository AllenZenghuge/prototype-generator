# Shot-Mode 根因分析与 Skill 质量审计

> 2026-07-03，基于深蓝汽车销项开票页面 ~15 轮迭代的经验总结

## 1. 问题总览

shot-mode 首次端到端测试从接手到满意共经历约 15 轮修正，涉及：

| 类别 | 轮次 | 典型问题 |
|------|------|---------|
| 列识别 | 4 轮 | 漏列（价税合计等 7 列）、列顺序调整、缺复选框+序号 |
| 顶栏 | 3 轮 | 双层结构、图标位置、重复元素 |
| 侧边栏 | 2 轮 | 菜单不完整、布局未全高 |
| 表格交互 | 2 轮 | 行操作图标 vs 文字、横向滚动 |
| 底部 | 2 轮 | 分页居中/同行、合计金额位置 |
| 筛选栏 | 1 轮 | 冻结 |

## 2. 根因分析

### 根因 1：视觉模型上下文丢失（最核心）

**现象**：Qwen3-VL 漏识别 7 列，误读操作列
**原因**：用户描述"前 10 列在截图 A，剩余列在截图 B"未传入 API prompt。模型收到通用 prompt，不知截图是宽表的左右分片。
**修复**：vision-analyzer.md 新增 Phase 0——从文件名+用户描述提取上下文，用户确认后注入 prompt。
**状态**：✅ 已修复

### 根因 2：缺少页面模板骨架

**现象**：侧边栏、顶栏、底部栏每轮都要调整
**原因**：规范.md 只有组件级的 CSS Token 和通用规则，没有「固定不变的页面骨架」约束。每次生成都是从头推断布局。
**修复**：规范.md 新增 §14「标准页面模板」，明确侧边栏全高/双层顶栏/sticky筛选栏/固定前列/底部合计分页同行等不变项。
**状态**：✅ 已修复

### 根因 3：多截图分析未传递上下文

**现象**：宽表后部分截图只看分页忽略列头
**原因**：两个截图独立发 API，第一张的分析结果（前 10 列）未传给第二张的 prompt。
**修复**：vision-analyzer Phase 0.4 规则——多截图时，先分析左部分，将结果写入续图的 prompt："已识别列 1-10 为：[...]，请识别后续列"。
**状态**：✅ 已修复

### 根因 4：结构化描述粒度不足

**现象**：行操作被识别为文字链接，实际是图标按钮
**原因**：通用 prompt 未区分「图标按钮」vs「文字链接」vs「混合」。
**修复**：prompt 模板增加了组件类型细分（button-icon / button-text / link-text）。
**状态**：⚠️ 部分修复，需在 ANALYSIS_PROMPT 中加入类型枚举

### 根因 5：增量修改导致代码残留

**现象**：出现双重图标（app-header-right 旧代码残留）
**原因**：多轮迭代中多次修改 framework.js，旧结构未清理。
**修复**：规范.md §14 固定骨架后，页面模板只需修改内容区，框架 JS 不再频繁变更。
**状态**：✅ 已修复（模板化后增量修改范围受限）

### 根因 6：缺少「生成后自检」步骤

**现象**：每轮发现问题都是用户反馈，非 skill 主动发现
**原因**：verifier.md 的检查清单未被 shot-mode prompt-builder 引用。生成后没有自动对照模板校验。
**修复**：prompt-builder 需在生成后读取 verifier.md 执行自检，对照规范.md §14 固定元素清单逐项验证。
**状态**：❌ 待修复（需更新 prompt-builder.md）
